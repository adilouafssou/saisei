# エンジニアリング規約 — Saisei（再生）エンジニアリングルールブック

Saisei 自律型早期警戒・経営改善計画書（ケイエイカイゼンケイカクショ）オーケストレーターの
永続的なメモリとルールブックです。

> Saisei は日本の地方銀行向けの融資後エージェントエンジンです。中小企業の財務健全性の悪化を検知し、
> ヒューマン・イン・ザ・ループの戦略交渉を通じて規制準拠の経営改善計画書の作成をオーケストレーションします。

---

## 1. 技術スタックの制約（ハード要件）

| コンポーネント | 制約 |
|---|---|
| Python | `>= 3.12` |
| LangGraph | `>= 0.2.x` |
| Reflex | `>= 0.6.x` |
| Pydantic | `V2 構文のみ` |
| パッケージマネージャー | `uv`（Astral）— `pip`/`poetry` を直接使用しないこと |
| ステートチェックポインティング | `psycopg`（v3）経由の PostgreSQL |
| キャッシュ / キュー | Redis |
| ロギング | `structlog` |
| リント / 型 / テスト | `ruff`、`mypy`、`pytest` |

## 2. 厳格な境界（交渉の余地なし）

- ロギングに `print()` を**絶対に使用しないこと**。常に `structlog` を使用すること。
- API キー、シークレット、DB 認証情報を**絶対にハードコードしないこと**。環境変数 / `pydantic-settings` を使用すること。
- **常に** Pydantic V2 構文（`model_config`、`field_validator`、`ConfigDict`）を使用すること。V1 の `class Config` / `@validator` は使用しないこと。
- 外部 API を**絶対に推測・ハルシネーションしないこと**。不明な場合は `mocks/` でモックすること。
- すべてのパブリック関数に型アノテーションを**常に付けること**。`mypy --strict` が通過しなければならない。
- 大規模なマルチファイルダンプを**絶対にコミットしないこと**。1 コンポーネントずつ変更し、検証してから進むこと。
- 元本金額には**常に** JPY を整数として保持すること。通貨元本に浮動小数点数を使用しないこと。
- 検証ループを**絶対にバイパスしないこと**。すべての変更には検証チェックリストを添付すること。

## 3. 日本金融ドメインルール（厳格に適用すること）

### 会計（J-GAAP）
試算表（しさんひょう）は標準的な日本の勘定科目を使用します：
- 売上（うりあげ）— 売上高
- 売上原価（うりあげげんか）— 売上原価（COGS）
- 販売費（はんばいひ）— 販売費及び一般管理費（SG&A）
- 経常利益（けいじょうりえき）— 経常利益

### 通貨とフォーマット
- JPY 元本 = 厳格な `int`（小数なし）。
- 表示フォーマット：日本式カンマ区切り、例：`¥150,000,000`。

### 企業識別
- TDB 企業コード：7 桁。
- 法人番号：13 桁。

### 会計年度カレンダー
- モックデータのデフォルトは 3 月 31 日終了の日本の会計年度（3 月決算）。

### 規制フレームワーク（金融庁 金融検査マニュアル）
債務者区分は必ず以下を使用すること：
- 正常（せいじょう）— Normal
- 要注意（ようちゅうい）— Substandard
- 要管理（ようかんり）— Doubtful

### マクロ・決済
- 日本銀行のマクロ変数を考慮すること：T+1/T+2 決済サイクルと資金繰りギャップ（資金繰り）に影響する金利引き上げ。

## 4. ステート管理ルール（LangGraph HITL）

- グラフステートがプロセス再起動後も生き残るよう Postgres チェックポインターを使用すること。
- ヒューマン・イン・ザ・ループの戦略交渉は `interrupt()` ノードを使用すること。
- 実行の再開は `Command(resume=...)` のみを通じて行うこと。チェックポイントステートをアウトオブバンドで変更しないこと。
- グラフステートは単一の Pydantic V2 モデルです。ノードは部分的な更新のみを返します。
- ステート内のすべての金額フィールドは `int`（JPY）です。バリデーターは該当する場合に非負の元本を強制します。

## 5. ワークフロー：カーパシーメソッド（仕様 → 検証器 → 環境）

1. 実装前に**仕様**を承認すること。
2. **検証器**：`make verify`（ruff + mypy + pytest）が自動化されたゲートです。CI はすべての MR でこれを実行します。
3. **環境**：完全にコンテナ化されています。`make setup` / `make run` で再現可能です。
4. MR ごとに 1 コンポーネント。各 MR の説明には検証チェックリストを含めること。

## 6. リポジトリ規約

- ブランチ命名：`spec/*`、`feat/*`、`fix/*`、`chore/*`。
- コンベンショナルコミット（`feat:`、`fix:`、`docs:`、`chore:`、`test:`）。
- リポジトリにシークレットなし。`.env.example` が必要な変数を文書化すること。
