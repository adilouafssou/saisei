# Saisei（再生）

[![CI](https://github.com/adilouafssou/saisei/actions/workflows/ci.yml/badge.svg?branch=main)](https://github.com/adilouafssou/saisei/actions/workflows/ci.yml)
[![Python 3.12+](https://img.shields.io/badge/python-3.12%2B-blue.svg)](https://www.python.org/downloads/)
[![Ruff](https://img.shields.io/badge/lint-ruff-261230.svg)](https://github.com/astral-sh/ruff)
[![Checked with mypy](https://img.shields.io/badge/mypy-strict-2a6db2.svg)](https://mypy-lang.org/)

> **地方銀行向け中小企業再生支援のための自律型早期警戒・経営改善計画書作成オーケストレーター。**

Saisei は、中小企業借入人の財務健全性を監視し、金融検査マニュアルに基づく信用リスク区分を行い、
担当バンカーとの協働により規制準拠の経営改善計画書をヒューマン・イン・ザ・ループ方式で作成します。

**LangGraph**（ステートフルなマルチエージェントオーケストレーションとネイティブな HITL）、
**Reflex**（Python ネイティブ UI）、および厳格な **Pydantic V2 / mypy --strict** ドメインコアで
構築されています。

---

## 解決すべき課題

金融庁の**金融検査マニュアル**に基づき、地方銀行は各借入人を継続的に査定し、健全性が悪化した際には
融資を引き揚げるのではなく、**経営改善計画書の策定を支援する**ことが求められます。実務ではこの作業は
手作業で、時間がかかります。

- 担当者が毎月の試算表を目視で確認し、売上の落ち込み、マージン圧縮、価格転嫁の失敗などの兆候を探す。
- 信用リスク区分（正常／要注意／要管理）の適用が担当者によってばらつく。
- 日銀の金利正常化や T+1/T+2 決済圧力による資金繰りストレスが明示的にモデル化されることはほとんどない。
- 経営改善計画書そのものは、白紙から書き起こされる。

その結果、**早期警戒シグナルの把握が遅れ、中小企業がすでに経営危機に陥ってからようやく再生支援が届く**
という問題が生じています。

## ソリューション

Saisei は、担当バンカーが行うべき査定を端から端まで実行し、戦略を確定する前に
**バンカーの判断を仰ぎます**。

1. **インテーク（受付）** — 企業 ID を解決（7 桁の TDB 企業コード → 13 桁の法人番号）し、信用調査報告書を
   取得し、反社会的勢力チェックを実施。
2. **EWS スコアリング** — 毎月の J-GAAP 試算表のトレンドから 0〜100 の早期警戒シグナルを算出。
3. **マクロストレス** — 日銀の政策金利カーブと決済流動性を組み込み、資金繰りギャップを推計。
4. **FSA 区分** — シグナルを債務者区分にマッピング。正常先はモニタリングのみ、それ以外は再生ワークフローへ。
5. **戦略提案** — 価格転嫁、原価削減、販管費の合理化、資金繰り改善といった接地した戦略を生成。
   期待される改善幅は、架空の数字ではなく当該企業の**実際の**財務数値から導出されます。
6. **ヒューマン・イン・ザ・ループ** — グラフが**中断**し、バンカーが承認・修正要求・エスカレーションを選択。
7. **計画書作成** — 承認後、Markdown 形式で決定論的な計画書ドラフトを生成。オプションで LLM による
   文章改善パスを実行しますが、すべての数値は保持されます。

> **設計方針：** 数値は決定論的に計算され、それが唯一の正解です。LLM は文章の改善にのみ使用され、
> 数値を生成することは一切ありません。LLM が設定されていない状態でも、システムは完全に機能し、
> **オフラインでテスト可能**です。

## アーキテクチャ

共有 Pydantic V2 ステート上の単一 LangGraph `StateGraph` です。データ読み込みノードは
`MockDataProvider` インターフェースの背後にあり、決定論的なモックを実際のコアバンキング／TDB／EDINET
クライアントに**グラフを変更することなく**差し替えられます。

```
            START
              │
              ▼
          [intake] ── TDB code → Hojin Bango, profile, anti-social check
              │
              ▼
           [ews] ── monthly Shisanhyo → 0–100 Early Warning Signal
              │
              ▼
          [macro] ── BOJ rates + T+1/T+2 settlement → Shikin Kuri gap
              │
              ▼
        [classifier] ── FSA class: 正常 / 要注意 / 要管理
              │
              ▼
     [keieisha_hosho] ── 経営者保証 release score + succession readiness
              │            (runs for ALL borrowers — even healthy ones)
   ┌──────┴────── conditional edge (fsa_classification) ──────┐
   ▼ 正常 (Normal)                                  要注意 / 要管理 ▼
  END (monitor only)                                    [strategist] ◀───────────┐
                                                             │                    │
                                  ┌──────────fan-out──────────┼─────────┐        │
                                  ▼                           ▼          ▼        │
                       [main_bank_critic]        [sub_bank_critic]  [guarantor_critic]
                          (P1 accountability)     (P2 fairness)     (P0 compliance)
                                  └──────────fan-in───────────┼─────────┘        │
                                                             ▼                    │
                                                     [lead_arranger]              │
                                              (Torimatome: consensus +            │
                                               burden-sharing table)              │
                                                             │                    │
                                  ┌──── route_after_lead_arranger ────┐           │
                                  ▼ approved / needs_human            ▼ rejected  │
                                  │                          (revision_count<3) ──┘
                                  │                          (count≥3) → END (escalate)
                                  ▼
                       ╭─ interrupt() ─────────────╮
                       │ [hitl_negotiation]        │  ← banker reviews & decides
                       ╰───────────────────────╯  (the only real decider)
                                  │
           ┌──────────────┼────── Command(resume=...) ──┐
           ▼ approve           ▼ revise                 ▼ reject
     [plan_writer]        back to [strategist]      END (escalate)
           │
           ▼
   Keikakusho draft → (optional LLM polish) → END
```

`hitl_negotiation` を除く各ステップは決定論的なノードです。`hitl_negotiation` だけが唯一の真の
エージェントで、`interrupt()`/`Command(resume=...)` による一時停止を担います。3 つのクリティックと
`lead_arranger` は純粋なルールベースのゲートであり、判定や数値が LLM によって生成されることはありません。
`keieisha_hosho` はすべての借入人に対して実行され、クリティックと `lead_arranger` は要注意・要管理の
借入人に対してのみ実行されます。計画中の Part 4 マルチエージェント*シミュレーター*レイヤーについては
[`HANDOFF.md`](../HANDOFF.md) を参照してください。

### 主要な設計判断

| 懸念事項 | 決定 | 根拠 |
|---|---|---|
| オーケストレーション | LangGraph `StateGraph` | ファーストクラスのステート、条件付きエッジ、HITL 向けのネイティブ `interrupt()`/`Command(resume=...)` |
| ステート永続化 | Postgres チェックポインター | HITL の一時停止は数日に及ぶ場合があり、再起動後もステートを保持する必要がある |
| 金額 | カスタム `JPY` int 型 | 円の元本は整数のみ。型レベルで小数を拒否する |
| ドメインモデル | Pydantic V2、`frozen`、`extra="forbid"` | 不変かつクローズドな財務レコード。タイポや余分なフィールドは即座にエラーになる |
| FSA 区分 | クローズドな `StrEnum` | 規制上の区分は正確に 3 値のみ。型が 4 番目の値を不可能にする |
| LLM | オプション、文章改善のみ | 決定論性と監査可能性を最優先。モデルは数値を生成しない |
| データソース | `MockDataProvider` シーム | ライブクライアントを 1 つのインターフェースの背後に差し込める。グラフ変更不要 |

### スタック

| レイヤー | 技術 |
|---|---|
| フロントエンド | Reflex >= 0.6 |
| バックエンド / API | FastAPI + LangGraph >= 0.2 |
| ステート | PostgreSQL（psycopg v3 チェックポインター） |
| キャッシュ / キュー | Redis |
| エージェントメモリ | pgvector（長期記憶）＋ RediSearch（短期記憶） |
| ツール | uv、ruff、mypy（strict）、pytest、structlog |

### エージェントメモリ（アドバイザリー RAG）

フィージビリティ・クリティックは、検索された先例（過去の計画、ベンチマーク、金融検査マニュアルの該当箇所）で
**アドバイザリー専用**のノートを補強します。先例の想起は、スタックが既に稼働している Postgres と
 Redis 上の二層エージェントメモリとしてモデル化されます（新しいインフラは不要）。

- **長期記憶 → pgvector。** Postgres に埋め込まれた永続的な先例コーパス。網羅的で、再起動後も保持されます。
  `python -m app.backend.tools.retrieval_ingest` でシードします。
- **短期記憶 → RediSearch。** Redis 内の TTL 付き高速想起キャッシュで、クエリ時に書き込まれます。

検索はまず短期記憶を照会し、ミスした場合は長期記憶にフォールバックし、その結果を短期記憶に書き戻します。
各層は独立して任意です（`SAISEI_PGVECTOR_DSN` / `SAISEI_REDISEARCH_URL`）。どちらも設定されていない場合、
検索は決定論的なモックを使用し、システムは**オフラインでテスト可能**なままです。検索はアドバイザリー
専用であり、バンド、スコア、ゲート、ルーティングに影響を与えることはありません。

### 意図的なスコープ

- **厳格な型チェックは `backend` と `tests` をカバーし、`frontend` は対象外です。** Reflex は UI を
  動的な `Var` オブジェクトとしてモデル化し `mypy --strict` と相性が悪いため、価値のあるドメインコアを
  厳格にチェックします。
- **ストラテジストは透明なヒューリスティックを使用します**（例：3% の価格引き上げ、2% の原価削減）。
  これにより、提案されるすべての数値が説明可能で、企業の実績値に基づきます。学習進化については
  [`NEXT_STEPS.md`](今後のステップ.md) を参照してください。
- **債務者区分は FSA の 5 段階のうち 3 段階をモデル化しています**（再生エンジンが機能する「まだ救える」帯域）。
  詳細は [`DOMAIN_ONBOARDING.md`](ドメイン入門.md) を参照してください。

## クイックスタート

```bash
cp .env.example .env      # no LLM key required to run
make setup                # install uv, sync deps, build containers, seed DB
make seed-memory          # （任意）pgvector 長期記憶をシード
make run-dev              # web + api + postgres + redis
make verify               # ruff + mypy --strict + pytest (the CI gates)
```

主要なフィクスチャは、原価高騰と価格転嫁の失敗に苦しむ愛知県の金属部品メーカー
（愛知精密製作所株式会社）です。これにより要管理区分と資金繰り赤字が生じており、まさに再生ワークフローが
対処すべき典型的なケースです。

> **再現可能なビルド：** `make setup` は `uv lock` を実行します。生成された `uv.lock` を一度コミットすれば、
> 以降は CI と Docker イメージがバイト単位で同一の依存セットを解決します。

## 継続的インテグレーション

[GitHub Actions](.github/workflows/ci.yml) は `make verify` と同じゲート — **ruff**（リント＋フォーマット）、
**mypy --strict**、**pytest** — を、`main` への全プッシュと全プルリクエストで実行します。`uv` で依存関係を
インストールし、完全にオフライン（アプリは決定論的なモックプロバイダーにフォールバック）で動作するため、
CI にはデータベース、Redis、API キーは不要です。GitHub Actions が初めての方は
[`docs/GITHUB_CI_SETUP.txt`](../GITHUB_CI_SETUP.txt) の手順ガイドを参照してください。

## リポジトリマップ

```
app/
  main.py                     FastAPI /health /ready + rx.App() lifespan
  backend/
    state.py                  SaiseiState (Pydantic V2) + reducers + sub-models
    graph.py                  StateGraph wiring, routers, Postgres checkpointer
    agents/                   turnaround_orchestrator (HITL interrupt/resume)
    nodes/                    Deterministic workflow steps
      financial_extraction.py   intake (TDB identity, anti-social) + macro (Shikin Kuri gap)
      ews_scoring.py            EWS compute (0-100) + FSA classify
      kaizen_generation.py      strategist + plan render + optional LLM polish
      keieisha_hosho.py         guarantee-release + succession assessment
      lead_arranger.py          consensus engine (Torimatome fan-in)
      critics/                  3 parallel critic nodes (main_bank / sub_bank / guarantor)
    tools/                    MockDataProvider + fixtures/ (bundled JSON)
      retrieval.py              two-tier agent memory: pgvector (long-term) + RediSearch (short-term)
      retrieval_ingest.py       seed precedent corpus into pgvector long-term memory
    prompts/                  extraction_rules.md, kaizen_templates.md
  frontend/                   Reflex UI (state, components, pages)
  shared/
    constants.py              Single source of truth for all thresholds
    settings.py               pydantic-settings (env prefix SAISEI_)
    models/                   accounting (TrialBalance), money (JPY), classification (FsaClass)
assets/                       Static assets for Reflex (images, fonts, CSS overrides)
tests/                        pytest — domain, nodes, graph flow, end-to-end
```
