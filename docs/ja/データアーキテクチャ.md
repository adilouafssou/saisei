# 再生 (Saisei) — データ・メモリアーキテクチャ

> **対象読者。** 本書はエンジニアと非エンジニアの双方を対象としています。経営層、投資家、
> 規制当局、監査人、提携金融機関のいずれもが読めるよう、*あらゆる種類のデータがどこに
> 保存され、なぜそこに保存され、誰が読み取ることを許されているか、そして本設計が日本の
> 規制下にある金融プロダクトの要件をどのように満たすか* を説明します。
>
> **位置づけ。** 本書は本リポジトリに実装されているとおりのシステムを記述します。以下で
> 言及するすべての構成要素は `app/backend/` 配下の実在モジュールに対応します。記載内容は
> **2026年6月** 時点のものです。

---

## 1. エグゼクティブサマリー

Saisei は、日本の地域金融機関向けの早期警戒および経営改善計画オーケストレーターです。
中小企業（SME）借り手の財務健全性を評価し、金融庁（FSA）の枠組みに基づいて信用リスクを
分類し、銀行担当者を関与させながら規制対応の経営改善計画書を共同で作成します。

本システムは、性質の根本的に異なるデータを扱います。意思決定時に参照される判例知識、
借り手の財務数値、銀行担当者が記録した判断、そして長期的にシステムを改善するために用いる
履歴コーパスです。**これらは意図的に別々のストアに、別々の経路で、別々のアクセスパターンと
ライフサイクルで保持されます。** この分離は偶発的なものではなく、ガバナンスおよび信頼性上の
要件です。

設計全体を貫く3つの原則は次のとおりです。

1. **数値は決定論的に計算され、それが真実の源（source of truth）である。** 大規模言語モデル
   （LLM）は言語と要約のためにのみ用いられ、数値・信用分類・ゲート・ルートを生成したり
   改変したりすることは決してありません。すべての財務出力は再現可能であり、円単位で監査
   可能です。
2. **意思決定経路と学習経路は物理的に分離されている。** ライブエージェントが推奨を行うために
   読み取るデータは、システムが後で学習するために書き込む大量のデータとリソースを
   奪い合うことはありません。
3. **既定で安全（safe by default）。** 外部サービスが一切設定されていない状態でも、Saisei は
   決定論的なモックに対して完全にオフラインで動作します。各連携は明示的に設定された場合に
   のみ有効化され、いかなるエラー時にもオフライン動作へ安全に縮退します。

## 2. 二つの面（プレーン）：意思決定経路 対 データ・フライホイール

Saisei 内のすべてのデータは、ちょうど二つの面のいずれか一方に属します。

| | **ランタイム（意思決定経路）** | **オフライン（データ・フライホイール）** |
|---|---|---|
| **目的** | 銀行担当者への単一のライブ推奨を支援 | 長期的なシステム改善 |
| **タイミング** | 同期的、実行のクリティカルパス上 | 非同期的、クリティカルパス外 |
| **読み取りパターン** | 読み取り中心、オンライン、低遅延が要求される | バッチ、全件スキャン、オフライン |
| **書き込みパターン** | 稀（キャッシュのウォーミングのみ） | 追記専用、より大量 |
| **読み手** | ライブエージェント | アナリスト、評価ジョブ、較正ジョブ |
| **障害時の影響** | 推奨品質が低下するが、正確性は損なわれない | ライブ製品への影響なし |

この表から導かれる決定的な設計原則は次のとおりです。**学習データは書き込み中心・
オフライン読み取りであり、ライブエージェントが意思決定中に問い合わせることは決してない。**
二つの面を分離することで、大量の学習書き込みがライブの借り手評価を遅延させたり不安定化
させたりすることはあり得なくなります。

## 3. 意思決定経路上のエージェントメモリ

実現可能性クリティック（feasibility critic）が提案された経営改善戦略を評価する際、
関連する判例コンテキスト（過去の経営改善計画の抜粋、業界ベンチマーク、金融検査マニュアルの
一節）を想起します。この想起は **助言のみ（advisory only）** です。人間が読むためのノートを
充実させるだけであり、決定論的なスコア・バンド・ゲート・ルートに供給されることは
ありません。

判例の想起は人間の記憶になぞらえてモデル化され、プラットフォームが既に運用している
インフラの上に二層で配置されます。**この二層は同じ仕事をする二つのデータベースでは
ありません。耐久性のあるストアの前に高速キャッシュを置いた、一つのメモリシステムです。**

### 3.1 長期記憶 — 耐久性のある知識ベース（pgvector）

長期記憶は判例知識の真実の源です。システムが学習したすべての判例をベクトルとして埋め込み、
pgvector 拡張を用いて PostgreSQL に耐久的に永続化します。網羅的であり、再起動を生き延び、
権威ある情報源となります。想起は判例テーブルに対する正直なコサイン類似度検索です。

- **モジュール:** `app/backend/tools/retrieval.py`（`PgVectorLongTermMemory`）
- **取り込み:** `app/backend/tools/retrieval_ingest.py`（冪等・バッチ・オフライン）
- **テーブル:** `saisei_keikakusho_memory`
- **性質:** 耐久的、網羅的、クエリあたりは低速、真実の源。

### 3.2 短期記憶 — ホットな想起キャッシュ（RediSearch）

短期記憶は Redis 上の高速なインメモリ想起キャッシュであり、直近で触れた判例を保持します。
ホットかつ安価であるため最初に参照されます。エントリには有効期限（TTL、既定で1時間）が
付与され、*失効することが想定されています*。この失効こそが当該層を「短期」たらしめる特徴
です。キャッシュエントリを失っても、損なわれるのは遅延だけであり、正確性は決して損なわれ
ません。

- **モジュール:** `app/backend/tools/retrieval.py`（`RediSearchShortTermMemory`）
- **インデックス:** `saisei_keikakusho_stm`
- **性質:** 揮発的、高速、使い捨て可能、長期記憶の厳密な部分集合。

### 3.3 二層の連携方法（ストアの前のキャッシュ）

オーケストレーター（`TwoTierRetrievalProvider`）は、人間の想起とまったく同じように
両者を結び付けます。

1. **まず短期記憶を確認する。** 関連する判例が直近に参照されていれば、即座に返す。
2. キャッシュミス時に **長期記憶へフォールバック** する。これまでに学習したすべての
   耐久的コーパス。
3. **定着（consolidate）。** 長期記憶の結果を短期記憶へ書き戻し、次回の類似クエリが
   ホットに応答されるようにする。

**これは「長期記憶の前に作業記憶を置く」標準的なパターンを、既に運用中のインフラに
マッピングしたものです。重複はなく、第三のベクトルエンジンは不要です。** RediSearch は
pgvector が耐久的に保持するものの、時間制限付き部分集合を保持します。Redis は耐久性を
犠牲に遅延を取り、PostgreSQL は遅延を犠牲に耐久性と網羅性を取ります。

各層はベストエフォートです。いずれの層も設定されていない場合、想起は判例を返さない
決定論的モックへフォールバックするため、ワークフローは完全にオフラインで動作し、
サービスの欠如や障害によって壊れることはありません。

```text
┌─ ランタイム（オンライン、意思決定経路上）─────────────────────────────┐
│                                                                      │
│  銀行担当者クエリ ─▶ feasibility_critic.search(query, top_k)          │
│                       │                                              │
│                       ▼  TwoTierRetrievalProvider                    │
│     1. 短期   RediSearch (TTL 約1h) ── ヒット ─▶ 返却（ホット, 約ms） │
│                       │ ミス                                         │
│     2. 長期   pgvector (耐久) ──────── ヒット ─▶ 返却                 │
│                       │                  └─ 3. 書き戻し ▶ RediSearch  │
│  (いずれも未設定) ─▶ モックプロバイダ ─▶ [] （オフライン安全）        │
└──────────────────────────────────────────────────────────────────────┘

   取り込み（オフライン・バッチ）: シードコーパス / 既存計画 / 検査マニュアル
       └─▶ retrieval_ingest.py ─ 埋め込み ─▶ pgvector (saisei_keikakusho_memory)
```

## 4. 借り手の実行状態と耐久的実行

一件の借り手評価は、銀行担当者の判断を待つ間、数時間にわたって一時停止することが
あります。その一回の実行の状態（数値、提案された戦略、途中で記録されたエラー）は、
LangGraph のチェックポインターによってセッションごとのスレッド識別子をキーとして
PostgreSQL に永続化されます。これが Human-in-the-Loop の中断を耐久的にしている仕組みです。
プロセスが再起動しても、実行は一時停止した地点から正確に継続します。

- **設定:** `SAISEI_POSTGRES_DSN`（`app/shared/settings.py`）
- **性質:** 耐久的な実行ごとの状態。中断・再開・監査の基盤。

## 5. データ・フライホイール（オフライン、意思決定経路には決して乗らない）

フライホイールは Saisei が改善していく仕組みです。何が起こり、人間が何を判断したかを
記録し、その記録をシステム調整のための根拠へと変換します。これらはいずれもライブの
意思決定経路上では実行されません。

### 5.1 実行トラジェクトリ — LangSmith

すべてのグラフ実行は、オフラインの分析と評価のために完全なトレースを LangSmith へ
送出できます。トラジェクトリは追記専用・時系列であり、アナリストや評価ジョブによって
オフラインで読まれます。ライブエージェントが再度読むことはありません。トレースは厳密に
オプトインです。LangSmith キーが未設定であれば、トレースは送出されず、ネットワーク呼び出しも
行われません。

- **モジュール:** `app/backend/observability.py`（`configure_tracing`）
- **性質:** 書き込み中心、オフライン、オプトイン。

### 5.2 人間の判断 — ラベル付き結果コーパス

将来の評価と調整のための原材料として、二種類の人間の判断を記録します。

- **HITL 銀行担当者の判断**（承認 / 修正 / 却下）は、`capture_hitl_feedback` により
  LangSmith データセット（`saisei-hitl-decisions`）にラベル付き事例として記録されます。
  厳密にオプトインであり、トレース未設定時は何もしません（ノーオペ）。
- **照合（reconciliation）結果** — 表面化した不一致が真に意味のあるものだったかという
  「どちらが正しかったか」の記録 — は実行状態（`reconciliation_outcomes`）に追記され、
  チェックポインターと共に PostgreSQL に永続化されます。

- **性質:** 追記専用、低量、人手によるラベル付け、オフラインジョブがバッチで読む。

### 5.3 根拠に基づく較正（キャリブレーション）

結果コーパスには具体的な見返りがあります。記録された人間の判断を、ある感度の高い閾値
（照合バンド距離）に対する **助言的** な推奨へと変換し、手で選んだ定数を根拠に基づく
値へ置き換えます。較正は純粋・決定論的・オフラインかつ助言のみであり、レポートを生成する
だけで、定数・ゲート・ルートを編集することはなく、LLM を呼び出すこともありません。

- **モジュール:** `app/backend/analysis/threshold_calibration.py`（純粋な分析）、
  `app/backend/analysis/calibrate_cli.py`（`make calibrate`）
- **性質:** バッチ、全件スキャン、オフライン、再現可能。

```text
┌─ オフライン（データ・フライホイール、意思決定経路には決して乗らない）─┐
│  各実行           ─▶ LangGraph ─送出─▶ LangSmith トレース（オプトイン）│
│  HITL 判断        ─▶ capture_hitl_feedback ─▶ LangSmith データセット    │
│  照合結果         ─▶ reconciliation_outcomes ─▶ Postgres（実行状態）    │
│                          └─▶ calibrate_cli ─▶ 助言的な閾値レポート       │
│                          └─▶ ［将来］学習 / ファインチューニング用コーパス│
└──────────────────────────────────────────────────────────────────────┘
```

## 6. 役割の一覧

各ストアの役割はちょうど一つずつであり、互いに重複しません。

| ストア | 面 | 役割 | ライフサイクル |
|---|---|---|---|
| **pgvector**（`saisei_keikakusho_memory`） | ランタイム | 耐久的な RAG の真実の源（長期記憶） | 永続、蓄積 |
| **RediSearch**（`saisei_keikakusho_stm`） | ランタイム | pgvector 上のホットな想起キャッシュ（短期記憶） | 揮発、TTL 付き |
| **PostgreSQL チェックポインター** | 双方 | 耐久的な実行ごとの状態。照合結果コーパス | 永続、追記専用 |
| **LangSmith** | オフライン | トラジェクトリと判断の記録 | 追記専用、オプトイン |
| **較正分析** | オフライン | 助言的な閾値推奨 | バッチ、導出 |

## 7. 設計判断：フライホイールデータ専用のベクトルストアは設けない

しばしば問われるのは、フライホイールデータ（トラジェクトリと学習データ）を、それ専用の
組み込み型あるいはファイルベースのベクトルストアに置くべきか、という点です。**本番システムに
おいては、答えは「いいえ」です。** 理由は三つあります。

1. **そのクエリはベクトル検索を必要としない。** フライホイールデータは追記専用であり、
   全件スキャン（「日付 X 以降のラベル付き結果すべて」）で読まれます。最近傍検索では
   ありません。ベクトルストアは、このデータに対して実行しないクエリのために最適化する
   ことになります。
2. **利点なしにガバナンス上の対象範囲を増やす。** 規制下の金融では、ストアが増えるごとに、
   保護・バックアップ・監査での説明が必要なものが一つ増えます。ここでの利点は仮説的で
   あり、コストは現実です。
3. **耐久的な基盤は既に存在する。** コーパスと実行状態には PostgreSQL、トレースには
   LangSmith、ホットな想起には Redis。専用ベクトルストアは隙間を埋めるのではなく、
   pgvector と重複するだけです。

**唯一許容される例外は開発者向けツールです。** エンジニアが、本番システムの完全に外側で、
ノート PC 上でエクスポートしたトラジェクトリに対してアドホックな意味検索を行う場合です。
これはランタイムの中核（spine）からは厳密に切り離されたままにします。

## 8. データガバナンスの姿勢

- **決定論的な真実の源。** すべての財務数値・信用分類・ゲート・ルートは、LLM ではなく
  監査可能な Python によって生成されます。モデルは言語に限定され、数値の改変は禁じられて
  います。
- **面ごとの最小権限。** ライブエージェントは推奨を行うために必要なものだけを読みます。
  学習データはオフライン面に隔離され、意思決定経路上で読まれることはありません。
- **助言の境界は明示的。** 判例の想起と閾値較正は、構成上、助言的であり、決定論的な
  結果を変えることはできません。
- **既定で安全、設定により明示的に有効化。** すべての外部連携（LLM、埋め込み、pgvector、
  RediSearch、BOJ、国税庁法人番号、LangSmith）は、設定されるまで無効であり、エラー時には
  決定論的なオフライン動作へ安全に縮退します。本システムは外部依存ゼロで完全に実行・
  テスト可能です。
- **ソースに秘密情報を持たない。** すべての設定は型付き設定を介して環境から供給され、
  認証情報がコミットされることはありません。
- **端から端までの監査可能性。** 実行ごとの状態は耐久的にチェックポイントされ、構造化
  ログがすべてのノードイベントを記録し、オプトインのトレースが完全なトラジェクトリ記録を
  提供します。

## 9. まとめ

Saisei のデータアーキテクチャは、規制下の金融プロダクトとして意図的に保守的です。
エージェントメモリには耐久的なストアの前にキャッシュを置き、ライブの意思決定経路を
オフラインの学習経路から厳密に分離し、すべての数値を決定論的に計算し、各外部システムを
明示的な設定時にのみ有効化します。その結果、すべてのデータフローを、エンジニアにも、
監査人にも、規制当局にも、投資家にも、同じ一枚の図で説明できるシステムが得られます。
