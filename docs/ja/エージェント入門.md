# エージェント入門 — Saisei を通じて学ぶエージェント型 AI

> **本番グレードのエージェント型システム**を構築する AI エンジニア向けのガイド付きカリキュラムです。
> このコードベースを実例として使用します。知識は **2026 年 6 月**時点のものです。
>
> ソースコードを開きながら上から下へ読んでください。すべての概念は実際に実行できるファイルを指しています。

---

## 0. 「エージェント型」とは何か（そして何でないか）

**エージェント**とは、目標に向けてループ内でアクションを取る LLM 駆動のシステムであり、
モデルの出力が次に実行されるステップに影響します。最もエージェント的でないものから最もエージェント的なものへのスペクトラム：

1. **単一プロンプト** — 1 回の呼び出し、1 つの回答。
2. **チェーン / ワークフロー** — 固定された呼び出しシーケンス（決定論的な制御フロー）。
3. **ルーター** — モデルが事前定義されたブランチから選択する。
4. **エージェント** — モデルがツール、メモリ、動的制御フローを持つループを駆動する。
5. **マルチエージェント** — 複数の専門化されたエージェントが協調する。多くの場合スーパーバイザーを持つ。

**2026 年の重要な教訓（業界が内面化したこと）：** *問題を解決できる最もエージェント的でない設計を使うこと。*
自由形式の自律性はデモでは印象的ですが、本番では脆弱です。
Saisei は意図的に**1 つのヒューマン・イン・ザ・ループ interrupt を持つ構造化グラフ**であり、
自由に動き回る自律エージェントではありません。規制を受ける金融業務では、すべてのステップが
説明可能でなければならないからです。**自律性をリスクに合わせること。**

> 🔎 このリポジトリでは：`backend/graph/builder.py` が制御フロー全体を 1 画面に収めています。
> システムができることとできないことを*読んで*理解できます。これは機能です。

## 1. ステートはエージェントの心臓部

エージェントが実行中に「知っている」すべてのことはステートに存在します。ステートモデルを正しく設計すれば
ノードはシンプルな純粋関数になります。間違えると永遠に戦い続けることになります。

**原則：**

- ステートを**型付きかつバリデーション済み**にする（ここでは：Pydantic V2、`extra="forbid"`）。
  タイポは静かな `None` ではなく大きなエラーになります。
- ノードは**部分的な更新**（プレーンな dict）を返し、ステート全体は返しません。
  これによりノードが組み合わせ可能になり、リデューサーが明示的になります。
- **派生値をステートから除外する**。計算します（`accounting.py` の `@computed_field` 利益行を参照）。
  派生データを保存することがステートが古くなる原因です。

> 🔎 学習：`backend/graph/state.py`（共有 `SaiseiState`）と、`backend/graph/nodes/` の各ノードが
> 自分が所有するフィールドのみを返す方法。

**演習：** EWS 出力に `confidence: float` フィールドを追加してみてください。ステートモデルと
1 つのノードだけを変更すれば済むことに気づくでしょう。他は何も壊れません。
これが良いステート設計の恩恵である疎結合です。

## 2. グラフ：ノード、エッジ、条件付きルーティング

グラフは制御フローを**データ**にします。ノードはステップ、エッジは遷移、条件付きエッジは
ノードの*結果*が次のステップを選択できるようにします。

- **線形エッジ**は決定論的な査定パイプライン用（`intake → ews → macro → classifier`）。
- **条件付きエッジ**は判断用：`route_after_classification` は健全な借入人の実行を終了し、
  そうでない場合はストラテジストに分岐します。
- **サイクル**は交渉用：`revise` はストラテジストに*戻る*ルートを取ります。ループを宣言的に表現します。

> 🔎 学習：`builder.py` の `route_after_classification` と `route_after_negotiation`。
> これらは小さな純粋関数であり、ユニットテストが簡単です（`tests/test_graph_flow.py`）。

**なぜ Python の `if` ではなくグラフなのか？** グラフは**チェックポイント境界**でもあり、
**可観測性境界**でもあるからです。各ノードは永続化、トレース、リトライ、再開の場所です。
次の 2 つのセクションがこれについて説明します。

## 3. ヒューマン・イン・ザ・ループ（HITL）：最も過小評価されているパターン

エージェントを実際のワークフローに展開するための最も高いレバレッジのパターンは、
**いつ止まって人間に尋ねるかを知ること**です。2026 年において、HITL は弱いモデルのフォールバックではなく、
高リスク・不可逆・規制を受けるアクションに対する正しい設計です。

Saisei は標準的なパターンを実装しています：

- `hitl_negotiation` ノードが **`interrupt(payload)`** を呼び出し、グラフを*一時停止*して
  提案された戦略をバンカーに提示します。
- 実行はアプリケーションがバンカーの判断を含む **`Command(resume={...})`** を呼び出したときにのみ再開されます。
- 人間が数時間かかる場合があるため、ステートは一時停止をまたいで**チェックポインターによって永続化**されます。
  プロセスが再起動しても実行は継続されます。

> 🔎 学習：`backend/graph/nodes/hitl_negotiation.py`（interrupt + 判断処理）と
> `frontend/saisei_ui/state.py`（`run_assessment` → 一時停止 → `approve/revise/reject` → 再開）。
> これはエンドツーエンドの完全な実際の HITL ループです。

**メンタルモデル：** interrupt は、戻り値が数分または数日後に人間から来る関数呼び出しです。
チェックポインターがそれを可能にするものです。

## 4. 永続化とメモリ

人々がよく混同する 2 つの異なるもの：

- **短期 / スレッドメモリ** = *1 回の実行*のステート。**チェックポインター**（ここでは：Postgres）によって
  `thread_id` をキーとして永続化されます。これが interrupt/resume とタイムトラベルを可能にします。
- **長期メモリ** = 実行をまたいで存在する知識（過去の計画書、埋め込み、ユーザー設定）。
  Saisei はまだこれを必要としません。`NEXT_STEPS.md` が RAG レイヤーとして追加します。

> 🔎 学習：`builder.py` の `postgres_checkpointer()` と、UI がセッションごとの `thread_id` を
> どのように渡すか。`MemorySaver` に差し替えると（テストがそうするように）、同じグラフがインメモリで動作します。

## 5. 決定論性、ツール、LLM の役割

2026 年の本番の教訓（業界全体で苦労して学んだこと）：**決定論的なコードがより得意なことを
LLM にやらせないこと。** 算術、明確な閾値を持つ分類、フォーマットはコードの仕事です。
LLM は言語、曖昧さ、統合のためのものです。

Saisei はこれを体現しています：

- EWS スコアリング、資金繰り推計、FSA 区分は**プレーンな Python**（`ews_agent.py`、
  `macro_agent.py`、`classifier.py`）— 円単位で監査可能かつテスト可能。
- LLM は**1 回だけ**、最終文書のオプションの文章改善パス（`backend/graph/polish.py`）として登場し、
  数値の変更を明示的に禁止されています。
- LLM が設定されていない場合は**no-op に劣化**するため、システムはオフラインで完全にテスト可能であり、
  不安定な API でも壊れません。

> 🔎 学習：`polish_keikakusho` — 任意の失敗時に決定論的なドラフトを返す best-effort の
> try/except に注目してください。この「LLM は強化として、依存関係としてではなく」というスタンスが
> 信頼性の高いエージェントを出荷する方法です。

**ツール呼び出しの注意：** ここでの「ツール」は型付きデータプロバイダーメソッド
（`MockDataProvider.credit_report`、`.shisanhyo`、...）です。ツールが LLM によって呼び出されるか
グラフに配線されるかにかかわらず、同じルールが適用されます：**型付きの入出力、境界でのバリデーション、
明示的なエラー処理。** 不明な企業の KeyError はクラッシュではなくステートに記録されたエラーになります
（`intake_node`）。

## 6. 信頼性：エラー、冪等性、データシーム

- **エラーはステートとして扱い、実行を終了させる例外としてではなく。** ノードは `state.errors` に
  追記し、適切な場合は継続します（`intake_node`、`ews_node` を参照）。グラフは検査可能なままです。
- **クリーンな統合シーム。** すべての外部呼び出しは `MockDataProvider` を通じて行われるため、
  システム全体がオフラインでテスト可能であり、ライブクライアントはグラフ変更ゼロで差し込めます
  （`mocks/provider.py`）。このシームを早期に設計することが、`NEXT_STEPS.md` のセクション 3 を
  書き直しではなく設定変更にするものです。
- **構造化ログをどこでも**（`structlog`、`print` は禁止）。各ノードは型付きイベント
  （`ews.scored`、`hitl.approved`）を出力します。これがトレースとメトリクスになります。

## 7. 評価と可観測性（2026 年の本当の仕事）

エージェントの構築は仕事の約 30% です。*すべての変更後もまだ動作することを知ること*が残りです。
デモと製品を分ける規律：

- **決定論的なコアを徹底的にユニットテストする**（このリポジトリ：金額、会計、分類器、モック）。
- **インメモリチェックポインターでグラフの制御フローをテストする**。完全な interrupt/resume サイクルを含む
  （`tests/test_graph_flow.py`）。
- **LLM ステップにトレースベースの評価を追加する**：ゴールデンデータセット、安価な決定論的事前チェック付きの
  LLM-as-judge、CI での回帰ゲート（`NEXT_STEPS.md` §1 を参照）。
- **軌跡をキャプチャする** — すべての実行とすべての人間の判断 — 監査証跡と将来のトレーニングデータの両方として
  （`NEXT_STEPS.md` §2）。データフライホイールが競争優位です。

## 8. マルチエージェント：いつ使うか、いつ使わないか

マルチエージェントシステムはトークン、レイテンシ、デバッグサーフェスの点で**高コスト**です。
2026 年のコンセンサス：タスクが真に並列であるか、明確に専門化されたスキル・コンテキストを必要とする場合にのみ
マルチエージェントを検討すること。専門化されたサブエージェントを調整するスーパーバイザーは、
汎用エージェントの群れより優れています。

Saisei は現在シングルグラフです。自然なマルチエージェントへの進化（`NEXT_STEPS.md` 内）は、
バンカーが見る前にストラテジストの提案に異議を唱える独立した**実現可能性クリティックエージェント**です。
これは古典的な*生成器/批評者*ペアであり、コストに見合う数少ないマルチエージェントパターンの 1 つです。

## このリポジトリを通じた推奨学習パス

1. `backend/domain/money.py` + `tests/test_money.py` — 正しく行われた型付きドメインプリミティブ。
2. `backend/domain/accounting.py` — 計算された（保存されない）値を持つ不変レコード。
3. `backend/graph/state.py` — エージェントを結びつける共有ステート。
4. `backend/graph/nodes/*.py` — パイプライン順に読む。各ノードが部分的な更新を返すことに注目。
5. `backend/graph/builder.py` — 制御フロー全体、エッジ、チェックポインターを 1 ファイルで確認。
6. `backend/graph/nodes/hitl_negotiation.py` — システムの interrupt/resume の心臓部。
7. `tests/test_graph_flow.py` — エージェント全体が実行、一時停止、再開するのをテストで観察。
8. `frontend/saisei_ui/state.py` — UI が本番で中断可能なグラフをどのように駆動するか。
9. `backend/graph/polish.py` — 規律ある、オプションの、決して重要でない LLM の使用。
10. `NEXT_STEPS.md` — 上記すべてが実際の製品にどのようにスケールするか。

## 持ち帰るべきコアメンタルモデル

- **機能する最小の自律性。** 構造は本番では自由に動き回るエージェントより優れています。
- **ステートがアーキテクチャです。** 型付けし、バリデーションし、派生データを除外します。
- **グラフは制御フローをデータにします** — チェックポイント、トレース、再開ポイントを提供します。
- **高リスク・不可逆のアクションでは止まって人間に尋ねること。** HITL はパッチではなく、ファーストクラスの設計です。
- **決定論的なコア、エッジの LLM。** モデルは支援します。正確性において決して重要ではありません。
- **エラーはステートです。** 実行を検査可能に保ちます。
- **統合シームを早期に設計すること。** 書き直しと設定変更の違いです。
- **評価と軌跡キャプチャが製品です**。後付けではありません。
