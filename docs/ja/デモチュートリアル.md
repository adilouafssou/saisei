# Saisei (再生) — デモチュートリアル

> **対象読者:** Saisei のライブデモを実行し、表示内容を理解したい評価者・実務者。すなわち、金融機関の与信・リスク部門、技術・セキュリティのレビュアー、デューデリジェンスを行うパートナーを想定しています。日本の金融規制やエージェンティック AI に関する事前知識は不要です。所要時間は通して約 **10 分** です。

---

## 1. この製品が行うこと（平易な言葉で）

日本の地域銀行は、金融庁の規制により、融資実行後も各中小企業の借り手を継続的にモニタリングし、経営が悪化し始めたときには貸し剥がしではなく **事業の再生を支援** することが求められています。現在、この作業は担当者が手作業で行っています。月次の試算表を睦んで読み、リスクを推量し、白紙の状態から再生計画を作成します。問題が見つかる頃には、企業はすでに深刻な経営難に陥っていることが多いのです。

**Saisei は、この「見守り」と「再生計画の初稿作成」を自動化しながら、すべての実質的な意思決定は担当者の手に残す、エージェンティック AI システムです。** LangGraph でオーケストレーションされたマルチエージェントワークフローが、借り手の月次数値を読み、早期警戒スコアを算出し、規制当局と同じように与信リスクを分類し、企業自身の数値に基づいた具体的な再生戦略を推論し、独立した AI エージェントが各債権者の立場から計画を議論する「模擬債権者会議」を実行します。担当者が承認した後にのみ、正式な再生計画書を作成します。

革新性は、**「知能をどこに適用するか」** にあります。規制下の与信判断において、数値を捧造する AI は使い物になりません。Saisei はこの二つを分離します。すなわち、**AI エージェントは推論・オーケストレーション・起草を担い、すべての数値は決定論的（デターミニスティック）に計算される** のです。これにより、銀行検査官は各数値を正確に再現でき、生成モデルは数値を変更することが構造的に禁止されます。エージェンティックシステムの判断力と迅速さを、規制当局が求める監査可能性とともに得られます。

## 2. 始める前に

| 要件 | 備考 |
|---|---|
| **Docker** がインストール・起動された PC | macOS、Linux、または Windows（WSL2）。 |
| **`make`** が利用可能 | macOS/Linux にはプリインストール。Windows では WSL2 を使用。 |
| 初回セットアップ時のみのインターネット接続 | 依存関係とコンテナイメージのダウンロード用。 |
| **生成 LLM の API キー** | **任意。** エージェンティックエンジン―オーケストレーション、スコアリング、分類、戦略推論、模擬債権者会議―はキーなしで、決定論的なデモデータに対して動作します。キーが必要なのは、最終計画に対する任意の自然言語の**文章推敢（ポリッシュ）**を有効化する場合のみで、数値や意思決定には一切影響しません。 |

> **評価者向けの要点:** フルなエージェンティックワークフローが、**外部 API キー不要、クラウドアカウント不要、借り手データを環境外に出さずに** ご自身の PC 上で動作します。決定論的なコアによりデモは再現可能であり、銀行のセキュリティ・コンプライアンス部門が最初に問うこと―*データは社外に出るのか、結果を監査できるのか*―に初期状態で明確に答えます。

## 3. セットアップと起動（コピー＆ペースト）

リポジトリのクリーンなチェックアウトから：

```bash
# 1. テンプレートからローカル設定を作成（デモに秘密情報は不要）
cp .env.example .env

# 2. すべてをインストールし、コンテナをビルドし、DB を起動
make setup

# 3. フルスタックを起動（Web UI + API + Postgres + Redis）
make run-dev
```

スタックが起動したら、ブラウザで Web UI を開きます：

```
http://localhost:3000
```

以上です。`make setup` や `make run-dev` で問題が生じた場合は、[トラブルシューティング](#7-トラブルシューティング--faq) をご覧ください。

> **公開デモリンク向けのヒント:** Saisei は **データベースなし**（状態をメモリに保持）で、かつ **単一の共有パスワード** の背後でも動作でき、数名の関係者向けの一時的なデモをホストするのに適しています。`.env` で `SAISEI_PERSIST_CHECKPOINTS=false` と `SAISEI_DEMO_PASSWORD=<任意>` を設定してください。ローカル PC のデモではどちらも無視して構いません。

## 4. デモのストーリー — 愛知の製造業者

製品には現実的な組み込みケースが同梱されています：**愛知精密製作所株式会社**。愛知県の金属部品メーカー、創業 1978 年、従業員 84 名。直近 12 ヶ月で売上が低下しコストが上昇した―いわば「価格転嫁の失敗」による圧迫―ため、本製品が対処するために存在する経営難区分へと向かっています。

### ステップ 1 — 借り手を入力

UI で、デモ企業の **7 桁の TDB コード** を入力します：

```
1234567
```

アセスメントを開始します。Saisei は企業の同一性を解決し、反社会的勢力チェックを実行し、12 ヶ月分の試算表を読み込みます。

### ステップ 2 — アセスメントの実行を見守る

インターフェースは各ステップを完了ごとにストリーミング表示します。順に以下が表示されます：

| 段階 | 画面に表示されるもの | 意味 |
|---|---|---|
| **EWS スコア** | 0–100 の早期警戒シグナル（高いほど悪い） | 売上と利益率の悪化傾向を 1 つの数値に蒸留。 |
| **資金繰りギャップ** | 円単位の数値（資金繰り） | 日銀の金利圧迫を含めたキャッシュフローのストレス。 |
| **金融庁分類** | 要管理などの債務者区分 | 規制当局型のリスク階層。健全な企業はここで停止（モニタリングのみ）、経営難の企業は継続。 |

### ステップ 3 — 模擬債権者会議

経営難の借り手に対し、Saisei は具体的な **再生戦略** を提案します―例：価格転嫁、売上原価削減、販管費の合理化、資金繰りの改善―いずれも企業の *実際の* 数値から導出した利益改善額を伴います。

次に、三つの独立した「クリティック（批評者）」の声が、それぞれ債権者の役割を演じて計画を判断します：

- **主幹事銀行** — 説明責任：経営コミットメントは整っているか。
- **協調融資銀行** — 公平性：各行のステークに応じて負担が公平に分担されているか。
- **信用保証協会** — コンプライアンス：信頼できる 3–5 年の回復路線があるか。

**リードアレンジャー** がそれらの判定を単一の推奨と負担分担表に集約します―担当者が実際の会議の前に読むブリーフィングです。

### ステップ 4 — 人間の意思決定（最重要の瞬間）

これがデモで強調すべき部分です。**ワークフローは停止し、担当者であるあなたを待ちます。** 以下が可能です：

- **承認** — 選択した戦略にコミットし、計画を生成。
- **修正依頼** — メモ付きで差し戻し（例：経営コミットメントの確認）、戦略が再提案・再判定される。
- **エスカレーション / 却下** — 案件を引き継ぐ。

デモでは、経営コミットメントのフラグをオンにして **承認** してください。

> **なぜこれが重要か:** AI エージェントは推論し、議論し、推奨しますが、決して決定はしません。システムの構造が、何かがコミットされる前に人間が承認することを保証します―まさに規制が求めるものであり、自動化と説明責任の正しい役割分担です。

### ステップ 5 — 成果物

承認されると、Saisei は以下を生成します：

- **再生計画書（経営改善計画書）** — 担当者が編集・提出できる構造化された草案。
- **損益回復見通し（損益計画）** — 再計算された早期警戒スコアが「正常」領域に戻る月をハイライトした、月次のチャート。
- **エクスポート** — 計画を **Word（.docx）**、回復見通しを **Excel（.xlsx）** で。日本の銀行が実際にやり取りする形式。

これらの文書内のすべての円数値は、決定論的な計算からそのまま転記されています―再導出や捧造は一切ありません。

## 5. 他の組み込みケースを試す

Saisei には異なる結果を示せる複数のフィクスチャが同梱されています。別の借り手を入力すると、異なる経路をたどります：

| ケース | 示されるもの |
|---|---|
| 愛知の製造業者（TDB `1234567`） | 経営難 → 再生 → 計画のフルフロー（メインデモ）。 |
| 健全なサービス企業 | 「モニタリングのみ」の早期離脱―再生不要。 |
| 要注意の製造業者 | より穏やかな警戒階層。 |
| 資金繰り赤字の企業 | ギャップを押し上げる大きなキャッシュフローストレス。 |
| 深刻な経営難の製造業者 | 法的整理（ワークアウト）の境界付近のケース。 |

> 他のフィクスチャの正確なコードは `app/backend/tools/fixtures/` にあります。愛知ケース（`1234567`）が最初に示すべきものです。

## 6. 評価時に指摘すべき点

ステークホルダー向けにデモを実行する際に強調すべき 5 点：

1. **エージェンティックな知能と監査可能な数値の両立。** 複数の AI エージェントがオーケストレーション、リスク推論、模擬債権者議論を担う一方で、すべてのスコアと円数値は規制当局が再現できる決定論的な計算です。生成モデルは文章を改善するのみで、数値には触れられません。この組み合わせ―AI の判断力と証明可能な算術―が中核的な革新です。
2. **人間が構造的にループ内にいる。** 担当者の決定なしにシステムが戦略をコミットすることは *できません*―コンプライアンス要件がポリシー文書での約束ではなく、ワークフローに組み込まれています。
3. **ご自身の環境で動作し、必須の外部 API なし、借り手データを社外に出さない。** 金融機関のセキュリティ部門が最初に提起する懸念に直接答えます。
4. **日本の銀行業務をネイティブに理解。** 金融庁の債務者区分、経営者保証の解除、日銀の金利ストレス、経営改善計画書という成果物が、汎用的な転用ではなく一級の概念として扱われます。
5. **使うほど賢くなる―データフライホイール。** すべての承認 / 修正 / 却下がラベル付きデータとして捕捉され、実際の担当者の判断による独自のコーパスを経年で構築します。

## 7. トラブルシューティング / FAQ

| 症状 | 対処 |
|---|---|
| `make setup` がすぐに失敗する | 開始前に **Docker Desktop が起動している** ことを確認。 |
| `localhost:3000` に何も表示されない | スタックの起動完了まで 1 分ほど待ち、`make run-dev` のログで "ready" を確認。 |
| ポート 3000（または 8000）が使用中 | 他のプロセスを停止するか、`make stop` で以前の Saisei を停止。 |
| 「外部 / LLM の API キーは必要？」 | **デモでは不要。** フルなエージェンティックワークフローはキーなしで動作。`.env` の LLM 項目は空のままでよく、キーは最終計画の任意の文章推敢を有効化するのみで、数値には影響しません。 |
| すべてをリセットしてクリーンに始めたい | `make clean` でコンテナとボリュームを削除（`.env` は保持）。その後 `make setup` を再実行。 |
| データを失わずに停止したい | `make stop` でコンテナを停止し、DB ボリュームは保持。 |
| デモの数値はどこから来ている？ | `app/backend/tools/fixtures/` の同梱 JSON フィクスチャ―外部サービスは一切呼び出されません。 |

### 便利なコマンド

```bash
make run-dev   # フルスタックを起動（web + api + postgres + redis）
make stop      # コンテナを停止、データは保持
make clean     # コンテナ＋ボリュームを削除（.env は保持）
make setup     # 依存関係の（再）インストール、コンテナビルド、DB シード
```
