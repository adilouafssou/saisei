# 今後のステップ — プロトタイプから製品へ

Saisei は現在、クリーンで決定論的、かつ完全にテスト済みの**プロトタイプ**です。エージェントグラフは
エンドツーエンドで動作し、HITL ループも機能しており、オフラインでも完全に動作します。
この文書は、Saisei を**本番バンキング製品**へと発展させるための正直なロードマップです。
おおよそレバレッジの高い順に並べており、各項目には*なぜ*重要か、*具体的に何が必要か*を記載しています。

## 基本方針

- **決定論性が唯一の正解であり続けます。** すべての数値は計算・監査可能です。LLM は言語と推論を
  支援しますが、数値には関与しません。
- **すべてを観測可能に、すべてを再現可能に。** 規制を受ける金融機関は、事後にあらゆる判断を
  説明できなければなりません。
- **人間の権限は交渉の余地がありません。** バンカーが承認し、システムが提案します。

---

## 1. LLMOps と可観測性（LangSmith）

**なぜ：** LLM がループに入ると（戦略推論、計画書の文章改善）、「自分のマシンでは動いた」では
不十分です。ノードごとのトレース、評価、コスト、レイテンシが必要です。

**何をするか：**

- グラフに **LangSmith** トレーシング（`LANGCHAIN_TRACING_V2`）を組み込み、すべての実行に
  `tdb_code`、`thread_id`、FSA 区分、ノード名をタグ付けします。
- **LangSmith 評価スイート**を構築します：（財務プロファイル → 期待される FSA 区分、
  期待される戦略テーマ）のゴールデンデータセット。CI で実行し、回帰でマージをブロックします。
- 経営改善計画書の文章改善ステップに **LLM-as-judge** 評価を追加します：すべての数値が保持されているか？
  FSA 区分が逐語的に維持されているか？（数値保持チェックは judge の前に安価な決定論的ガードとして機能します。）
- ノードごとの**コストとレイテンシの予算**を追跡し、SLO を超えたときにアラートを出します。
- プロンプトをソースコードから**バージョン管理されたプロンプトレジストリ**（LangSmith Hub または
  内部テーブル）に移行し、プロンプトの変更がレビュー可能かつロールバック可能になるようにします。

## 2. エージェント軌跡データフライホイール

**なぜ：** バンカーのすべての判断（承認・修正・拒否 + コメント）は高品質な人間の選好ラベルです。
適切にキャプチャすれば、これがストラテジストを時間とともに測定可能に改善するトレーニングシグナルとなり、
防御可能な製品の核心になります。

**何をするか：**

- **完全な軌跡**を永続化します：入力ステート、各ノードの出力、interrupt ペイロード、
  バンカーの `Command(resume=...)`、最終計画書 — 追記専用ストア（Postgres + 大規模アーティファクト用
  オブジェクトストレージ）に保存します。
- 各交渉を**選好ペア**として扱います：承認された戦略（選択）vs. 代替案（拒否）、
  さらに修正コメントを批評として使用します。
- オフラインパイプラインを構築します：
  - 承認された戦略・計画書に対する小規模オープンモデルの**教師あり微調整**。
  - 承認・拒否ペアからの**選好最適化（DPO/ORPO）**。
  - 修正コメントからバンカーの承認を予測する**報酬モデルのトレーニング**。
- **オフラインリプレイ**でループを閉じます：過去の軌跡を候補モデルで再実行し、
  バンカーが実際に選んだ戦略をどれだけ提案できたかを測定します。
- 厳格な **PII・データガバナンス**境界：財務データは銀行の VPC から外に出ない。
  トレーニングはオンプレミスまたは専用テナンシーで実行します。

## 3. モックを実際の統合に置き換える

**なぜ：** アーキテクチャ全体が差し替え可能な `MockDataProvider` シームを中心に構築されています。
これを実現することが Saisei を本物にします。

**何をするか：**

- 既存のインターフェースの背後に `CoreBankingClient`、`TdbClient`、`EdinetMacroClient` を
  実際の API に対して実装します。グラフ変更不要。
- **レジリエンス**を追加します：バックオフ付きリトライ、サーキットブレーカー、Redis でのレスポンスキャッシュ、
  境界でのスキーマバリデーション（Pydantic モデルがすでに形状を強制しています）。
- ライブデータと期待される J-GAAP 不変条件（例：売上総利益 = 売上 − 売上原価）を比較する
  **照合テスト**で上流のデータドリフトを検出します。

## 4. エージェント推論の強化

**なぜ：** 現在のストラテジストは固定ヒューリスティック（3% の価格引き上げ、2% の原価削減、
5% の販管費削減）を使用しています。これは正しい*出発点*（根拠があり、説明可能）ですが、
実際の製品は業種や顧客集中度ごとの実現可能性を推論すべきです。

**何をするか：**

- 銀行の過去の成功した経営改善計画書、業界ベンチマーク、金融検査マニュアル自体に対する
  **検索レイヤー（RAG）**を追加し、提案が先例を引用できるようにします。
- 各戦略をストレステストする**実現可能性クリティック**ノード（第 2 のエージェント）を追加します
  （「サプライヤー集中型の企業が本当に原価を 2% 削減できるか？」）。バンカーに届く前に実行します。
- 単月の年換算ではなく**複数期間の予測**（12〜36 ヶ月の損益ブリッジ）をモデル化し、
  正常への回復経路を示します。

## 5. プラットフォームの本番化

**何をするか：**

- **API サーフェス：** 認証（OIDC）、銀行ごとのテナンシー、`thread_id` をキーとする
  冪等な実行・再開エンドポイントを備えた FastAPI でグラフを公開します。
- **非同期 + スケール：** 長時間の LLM/RAG 呼び出しをリクエストパスから外します。
  Redis/Celery（すでに依存関係）または LangGraph の非同期ランタイムを使用し、
  ステートレスワーカーを水平スケールします。
- **マイグレーション：** チェックポインターテーブルを超えた Postgres スキーマに Alembic を使用します。
- **シークレット：** Vault / クラウドシークレットマネージャー。コードベースはすでにハードコードされた
  シークレットを禁止しています。
- **エクスポート：** 銀行のテンプレートと電子署名フックを使用して経営改善計画書を PDF/DOCX にレンダリングします。

## 6. コンプライアンス、セキュリティ、監査

**なぜ：** これは規制を受ける融資業務です。信頼が製品です。

**何をするか：**

- すべての区分とすべての人間の判断（誰が、何を、いつ、どのデータバージョンで）の
  **不変監査ログ**。
- 各 FSA 区分に添付される**説明可能性レポート**（どのシグナルがどの閾値を超えたか）。
  決定論的なコアがこれを簡単にします。
- デプロイされたモデルの**モデルカードと変更ログ**。トレーニングに使用したデータスナップショットを記録します。
- 業種・地域をまたいだ**バイアス・公平性レビュー**。反社会的勢力チェックは監査可能かつ
  人間が上書き可能でなければなりません。

## 7. テストと品質の向上

- 金額型と会計不変条件に対する**プロパティベーステスト**（Hypothesis）。
- レンダリングされた経営改善計画書 Markdown の**スナップショットテスト**。
- 同時査定とチェックポインターの競合下での**負荷テスト**。
- スイートが実際に回帰を検出することを確認する**ミューテーションテスト**。

---

## 推奨実装順序

1. LangSmith トレーシング + CI でのゴールデン評価セット（安価で即効性のある安全網）。
2. 軌跡キャプチャ（必要になる*前*にデータ収集を開始する）。
3. 既存のシームの背後での 1 つのライブ統合（TDB）。
4. 実現可能性クリティックノード + 過去の計画書に対する RAG。
5. キャプチャした軌跡からの選好最適化。
6. コンプライアンス・監査の強化（全期間を通じて並行して実施）。
