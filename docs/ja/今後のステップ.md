# 今後のステップ — ロードマップ

これは前向きのバックログです。**次に何を作るべきか**を記したものであり、完了済みの記録ではありません。
システムの現状は [`README.md`](../../README.md) と、アーキテクチャ仕様である [`ROADMAP.md`](../en/ROADMAP.md)
を参照してください。過去の作業は、残作業を説明するために必要な範囲でのみ言及します。

本書は 3 部構成です：

1. **アクティブ・バックログ** — これから実装すべき作業。おおよそ優先順位順。
2. **保留中** — 構築済みだが、エビデンスが蓄積するまで製品判断で意図的に一時停止しているもの。
   記載されたトリガーが満たされるまで再開しないこと。
3. **V2 の地平線** — 次の次の方向性。意図的にアクティブ・バックログには含めず、ロードマップを
   無駄なく保つため仕様レベルより軽く記述しています。

4 つ目の短いセクション **運用上のフォローアップ** は、コード作業ではないライブエンドポイントの
確認項目を列挙します。各経路はすでにコード内でゲートされ、オフラインで green です。残るのは
実際のサービスへ実認証情報で接続することだけです。

## 基本方針

- **決定論性が唯一の正解であり続けます。** すべての数値は計算・監査可能です。LLM は言語と推論を
  支援しますが、数値には関与しません。
- **すべてを観測可能に、すべてを再現可能に。** 規制を受ける金融機関は、事後にあらゆる判断を
  説明できなければなりません。
- **人間の権限は交渉の余地がありません。** バンカーが承認し、システムが提案します。
- **より多くの能力を、しかし決して権限は与えない。** 新しい能力はバンカーに情報を提供してよいが、
  ゲート・ルート・数値において票を持ってはなりません。
- **検証されていない LLM の主張を、事実としてバンカーに届けてはなりません。** 数値は決定論的で
  検証ゲートに守られています。定性的な LLM 出力は、帰属可能な根拠（決定論的な数値、または検索された
  出典）に接地されるか、明確に「未検証」と表示されなければなりません。「助言のみ」は十分な安全策では
  ありません — 接地されていない論拠でも人間の判断を左右しうるからです。

## アクティブ・バックログ

**空です。** アクティブな*コード*バックログは解消されました。コードのシームそのものが成果物で
あったロードマップ項目はすべて出荷済みです（シークレットシーム、Alembic マイグレーション、監査
分析、そして PII セーフな VPC 内軌跡エクスポート境界）。かつてのアクティブ項目の残りは、本来
属する場所へ再分類しました：

- **運用であってコードではない** — RAG コーパスの拡充（実コーパスの取り込み + ソース単位の
  アクセス制御）、プラットフォームの本番化（マイグレーション実行、シークレットマネージャーの
  プロビジョニング、電子署名ベンダーの結線、分散実行ワーカー）、監査台帳のハードニング（物理削除
  + 外部公証）。検索／シークレット／マイグレーション／分析の*コード*経路は出荷済み・オフラインで
  green であり、残るのは実サービスへ向けることだけです。下記 *運用上のフォローアップ* を参照。
- **延期（コード、ゲート付き）** — 複数ラウンドの軌跡粒度。これが資する training ループと並べて
  *V2 の地平線* に記録しています。training 実行がラウンド単位のシグナルを必要とするまで作らないこと。
- **V2 の地平線** — PDF／写真 OCR 取り込み（残る唯一のキャプチャチャネル）。下記に既に記載済みで、
  確認・修正ループが堅牢化されるまで延期します。

ここに推測的な項目を追加しないこと。新しいアクティブ・バックログ項目は、成果物がコードである
（運用上の接続でも、製品判断による延期でもない）新鮮で・コード化可能で・未構築の作業があるときに
のみ追加すべきです。

## 運用上のフォローアップ（ライブエンドポイントの確認、コード作業ではない）

これらはロードマップ機能ではありません。各経路は出荷済み・オフラインで green・コード内でガード済み
（スキップガード／オフライン no-op 契約／`# VERIFY` ブランチ）であり、オペレーターが実行するまで
`make verify` はオフラインのままです。

- **LangSmith ゴールデンデータセットの push。** `observability.push_golden_dataset` を実際の
  LangSmith アカウントに対して実行し、`saisei-classification-golden` データセットが投入されることを
  確認する。
- **経営改善計画書の文章改善に対するライブ LLM-as-judge。** `tests/eval/online/test_polish_judge.py`
  を実際の LLM エンドポイントに対して実行する（決定論的な数値プレゲートはすでにオフラインで動作）。
- **実テレメトリからの閾値チューニング。** トレースが蓄積したら、`DEFAULT_FAITHFULNESS_FLOOR`
  （主張接地）と `analysis/node_budgets.py` のノードごとの `NODE_LATENCY_BUDGETS_MS` /
  コスト予算を観測値からチューニングする。
- **ライブデータクライアントの確認。** 各 `# VERIFY` ブランチ（TDB、コアバンキング、EDINET、BOJ、
  NTA 法人番号）を実サービスに対して確認し、ライブ読み取り経路に Redis レスポンスキャッシュを追加し、
  ネットワークマーク付きの統合テストを追加する。
- **OIDC プロバイダー登録。** トークン検証トランスポートは出荷済み・オフラインで green。残るのは
  運用作業：API を銀行の ID プロバイダー（Okta / Entra / Keycloak）にクライアントとして登録し、
  `SAISEI_AUTH_JWKS_URL` / `SAISEI_AUTH_ISSUER` / `SAISEI_AUTH_AUDIENCE` をその値に設定する。
  これらが未設定の場合、API はプレースホルダー ID の姿勢（`SAISEI_AUTH_REQUIRED` でゲート）を保ち、
  ネットワーク呼び出しを行いません。
- **データベースマイグレーションの実行。** Alembic がアプリケーションスキーマ（`alembic.ini` +
  `migrations/`；監査／軌跡／ポートフォリオ／pgvector メモリのテーブル。各ストアの `SCHEMA_SQL` を
  唯一の正解として再利用）を所有します。コード内の冪等なブートストラップは引き続き動作するため、
  新規クローンやオフライン実行ではマイグレーション手順は不要です。マネージドデプロイでは、リリースの
  一環として実際の `SAISEI_POSTGRES_DSN` に対して `make migrate`（Alembic `upgrade head`）を実行
  します。LangGraph チェックポインターテーブルは引き続き `PostgresSaver.setup()` が所有し、Alembic
  管理外です。
- **シークレットマネージャーのプロビジョニング + 認証情報。** Vault / クラウドシークレット
  マネージャーを立ち上げ、銀行ごとの実認証情報（DB DSN、LLM キー、データソース API キー）を投入
  します。コード内のプロバイダーシームがそれらを消費します（`app/backend/secrets.py`：シークレットは
  `resolve_secret` 経由で読まれ、`@env:` / `@file:` / `@/path` の参照は `.env` に触れず逆参照され、
  Vault / クラウドバックエンドは `set_secret_provider` で組み込まれます）。すべてのライブシークレット
  読み取りはこのシームを通ります（`postgres_dsn` / `audit_dsn`、`llm_api_key`（チャット + 埋め込み）、
  `tdb_api_key`、`core_banking_api_key`、`hojin_bango_app_id`、監査署名鍵 + LangSmith キー）。残るのは
  運用作業です。軌跡エクスポートについては、データガバナンス／PII レビュー後に
  `SAISEI_TRAJECTORY_EXPORT_ENABLED` + `SAISEI_TRAJECTORY_EXPORT_DIR`（および実際の
  `SAISEI_TRAJECTORY_EXPORT_SALT`、任意で `SAISEI_TRAJECTORY_EXPORT_INCLUDE_FREE_TEXT`）を設定し、
  選ばれたスレッドのエクスポートを `make export-trajectories ARGS="<thread-ids>"`（または
  `ARGS="--threads-file <file>"`；`--dry-run` を付けると書き込まずに件数をオフラインでプレビュー）
  で実行し、生成されたローカル JSONL を VPC 内の training 実行へ供給します。
- **実 RAG コーパスの取り込み。** 銀行の成功した経営改善計画書の蓄積データと金融検査マニュアル
  全文を pgvector に取り込み、ソース単位のアクセス制御を銀行の ID システムに結線します。取り込み
  ツールと助言のみの検索経路は既に存在します。実際のガバナンス対象コンテンツの読み込みは運用上の
  データガバナンス作業です。
- **電子署名の配信。** 電子署名プロバイダーと契約し、経営改善計画書のエクスポートを接続します
  （ベンダーアカウント、署名テンプレート、コールバック認証情報）。PDF/DOCX レンダリングは存在し、
  ベンダー結線はデプロイ／契約タスクです。
- **監査の物理削除 + 外部公証。** デプロイに委ねられた 2 つの特権的な監査台帳操作：(1) `plan_retention`
  からの保持対象スレッドの物理削除（専用の・個別ログ記録される DB ロールの下で追記専用トリガーを
  緩める必要がある。コード内の計画 + リーガルホールド除外は既に存在）；(2) 外部公証 — コード内の
  Ed25519 署名に加え、台帳ダイジェストを独立したタイムスタンプ機関／追記専用の外部ログに定期的に
  アンカーする。
- **分散実行ワーカー（水平スケール）。** リクエストパス外実行のシームはコードで出荷済み
  （`SAISEI_RUN_ASYNC` + プロセス内スレッドエグゼキューター；実行は `phase="running"` を返しポーリング
  される）。プロセスをまたいだスケールが運用ステップです：既存の Redis 上に分散ワーカー（Celery /
  RQ / Arq）を `RunExecutor` シームに組み込み、ワーカーレプリカを実行します。ルートのコードは変わ
  りません。プロセス横断の永続的な実行ステータスはワーカーと共に移動します。

## 保留中 — 照合閾値の較正ループ

LLM 対フロアの照合較正ループ（キャプチャ → 分析 → `make calibrate` → 表示パネル）は構築済みで、
キャプチャされた `reconciliation_outcomes` コーパスが有意になる（**ラベル付き成果 ≥ 10 件**）まで
製品判断で一時停止しています。シンプルな手設定の定数を維持し、キャプチャ工程にデータを蓄積させ、
十分なエビデンスが揃うまで分析／CLI／パネルの各層に手をつけないこと。再開する際は、HITL で
バンカーに「どちらが正しかったか」の判定を尋ね、次に推奨を**レビューし（決して自動適用しない）**、
最後に同じループを `MAX_RECONCILIATION_TRIGGERS` へ一般化します。

## V2 の地平線 — 摩擦のないキャプチャとトレーニング（MVP を超えて）

*アクティブ・バックログではありません。* 次の次の、製品を定義するアップグレード — コア
（査定 → リハーサル → 計画）と MVP のキャプチャ面が現場で実証された後の方向性です。ビジョンを
失わないために記録し、上記アクティブ・バックログより意図的に軽く記述します。すべてに同じ
アーキテクチャ上のコミットメントが適用されます：**各チャネルは同一の型付きレコードへ正規化し、
すべての非決定論的抽出は決定論的スパインに入る前にバンカーが確認する。**

- **Saisei への転送メール取り込み。** 借り手ごとの専用取り込みアドレス
  （例：`borrower-xxx@intake.saisei`）を用意し、顧問税理士が月次の試算表を自分の送信箱から
  そのまま送れるようにする。既存の確認・修正シームを通じて解析され、バンカー確認のためにキューされる。
  受信メールインフラ（アドレスの払い出し、添付ファイルの解析、スパム／不正対策と送信者検証の面、
  テナントごとのルーティング）が必要であり、アップロードとガイド付き入力が需要を実証して初めて
  価値を持つ。認証／ログインは明示的にこれではない（シングルサインオン／ユーザーアカウントは
  プラットフォーム本番化作業に属する）。これは純粋にゼロ手間の*データキャプチャ*チャネルである。
- **確認・修正付きの PDF／写真 OCR。** スキャンまたは撮影した試算表を投入すると、決定論的 OCR が
  数値を提案し、バンカーがソース画像と照合してセル単位で確認する。最もレバレッジの高いロングテール
  キャプチャ（困窮した中小企業のデータはしばしば紙にしか存在しない）であり、ドキュメント投入チャネルの
  自然な完成形。曖昧な抽出が確認なしに与信判断へ到達してはならないため、まず確認・修正ループを
  堅牢化する必要があり、ここで延期する。
- **選好最適化トレーニングループ（GRPO/DPO/ORPO + SFT + 報酬モデル）。** キャプチャした
  `(選択, 拒否, 批評)` の選好ペアを、測定可能により優れたストラテジストへと変える：教師あり微調整、
  選好最適化（GRPO/DPO/ORPO）、バンカーの修正コメントに対する報酬モデル。そして**オフラインリプレイ**で
  ループを閉じ、候補モデルがバンカーが実際に選んだ戦略をどれだけ提案できたかを測定する。
  アルゴリズム自体は今すぐ書けるが、その有効性は蓄積された実軌跡の量と、厳格な VPC 内
  エクスポート／トレーニング境界（財務データは銀行から外に出ない）に依存する。キャプチャシームは
  すでにトレーニングシグナルを生成しており、これはデータが存在して初めて価値を持つ消費側である。
  - **延期サブ項目（コード、ゲート付き）：複数ラウンドの軌跡粒度。** 現在は HITL 判断ごとに 1
    レコードをキャプチャする。各中間修正ラウンドを独立したレコードとしてキャプチャするのは今すぐ
    コード化可能だが、意図的に延期している — この training ループがラウンド単位のシグナルを必要と
    したときにのみ必要となる。単独ではなく、上記の消費側と一緒に（またはその直前に）構築すること。
- **スケジュール化された継続監視デーモン。** ポートフォリオ監視リストは永続化され、決定論的な
  読み取り側のプランニング（`portfolio/monitor.py`：期限到来の借り手向け `plan_refresh` + 閾値
  イベント向け `detect_crossings`）を備えている。V2 で残るのは、それらのプランナーを一定の頻度で
  駆動しバンカーに通知する**スケジュール化された取り込み + アラートデーモン**である — デプロイ
  インフラ（ジョブランナー + 通知チャネル）であり、その実データガバナンス判断（オンデマンドの 1 件では
  なく帳簿全体を保持する）の背後でゲートされる。Saisei の不変条件は保たれる：デーモンは可視化し通知
  するが、決して査定を自動実行しない。
