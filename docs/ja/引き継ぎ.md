# Saisei（再生）— エンジニア引き継ぎ資料

日本の中小企業再生（*再生ファイナンス*）向けの決定論的 LangGraph + Reflex エージェントエンジンです。
借入人の財務健全性を査定し、模擬債権者会議（協調融資銀行会議）を通じて経営改善計画書を審査し、
バンカーが**経営改善計画書**（ケイエイカイゼンケイカクショ）を作成するのを支援します。
この文書を一読すれば、すぐに開発に着手できます。

## すべてを支配する唯一のルール

**決定論性が唯一の正解です。LLM は文章を整えるだけであり、判定を下したり数値を生成したりすることは
決してありません。** すべてのスコア、ゲート、合否判定は純粋なルールベース関数です。
新機能を追加する場合、その判断は再現可能かつ監査可能でなければなりません。
LLM は文章の改善にのみ使用できます（`polish_keikakusho` を参照）。

必ず守るべき原則：
- 既存の `SaiseiState` フィールドを変更・削除しないこと（追加のみ可）。
- `render_keikakusho` の出力はバイト単位で同一であること（スナップショットに依存）。
- すべての金額は**厳格な整数円**（`app/shared/models/money.py`）。

## 90 秒でわかるアーキテクチャ

すべては統合された **`app/`** パッケージ配下にあります（Reflex のコンパイラが要求するため。
`rxconfig.py` で `app_name="app"` を設定）。

```
app/
  main.py                     # rx.App() + FastAPI /health /ready + lifespan
  backend/
    state.py                  # SaiseiState (Pydantic V2) + reducers + sub-models
    graph.py                  # StateGraph wiring + routers + Postgres checkpointer
    agents/                   # TRUE agents (loops / routing / tool-use / HITL)
      turnaround_orchestrator.py   # the ONLY agent: interrupt()/resume HITL
    nodes/                    # WORKFLOW steps (deterministic, pure-ish functions)
      financial_extraction.py # intake + macro (TDB identity, Shikin Kuri gap)
      ews_scoring.py          # EWS compute + FSA classify
      kaizen_generation.py    # strategist + plan render + LLM polish
      keieisha_hosho.py       # guarantee-release + succession assessment
      lead_arranger.py        # consensus engine (Torimatome)
      critics/                # 3 parallel critic nodes (main_bank/sub_bank/guarantor)
    tools/                    # swappable data clients (mocks today) + fixtures/
    prompts/                  # extraction_rules.md, kaizen_templates.md (static assets)
  frontend/                   # Reflex UI (state, components, pages)
  shared/                     # models/ (accounting, money, classification),
                              # constants.py, settings.py, logging.py
tests/                        # pytest; import canonically from app.*
```

**ノードとエージェントの区別**は設計上の核心です（混同しないこと）。*ノード*は決定論的な
単一パス関数であり、*エージェント*はループ・ルーティング・ツール使用または HITL を駆動します。
現在、エージェントは `turnaround_orchestrator` の 1 つだけです。

## グラフフロー（実際に実行される処理）

```
intake → ews → macro → classifier → keieisha_hosho
                                          │
              route_after_classification  ▼
   Joyo (正常) → END (monitor only)   |   Yoi Kanri / Yukyo Guchi → strategist
                                          │
   strategist ─fan-out─▶ main_bank_critic ┐
              ─fan-out─▶ sub_bank_critic  ├─▶ lead_arranger
              ─fan-out─▶ guarantor_critic ┘        │ route_after_lead_arranger
                                                   ├ approved / needs_human → hitl_negotiation
                                                   ├ rejected & count<3     → strategist (cycle)
                                                   └ rejected & count>=3    → END (escalate)
   hitl_negotiation (interrupt) ─▶ approve → plan_writer → END
                                  revise  → strategist
                                  reject  → END
```

- **keieisha_hosho はすべての借入人に対して実行されます**（健全先も含む）。クリティックは
  **要注意・要管理先のみ**に対して実行されます。
- ステートは HITL の `interrupt()` をまたいで **Postgres チェックポインター**によって永続化されます。
  `Command(resume={"decision": ...})` で再開します。

## 機能 1 — 経営者保証（ケイエイシャホショ）保証解除

`keieisha_hosho.py` は、FSA ガイドラインの 3 つの柱に基づき、すべて決定論的に解除適格性スコア（0〜100）を算出します。

- **法人個人分離**（分離、40 点）— `eigai_shueki / eigai_hiyo ≥ 1.0` プロキシ。
- **財務基盤の強化**（財務基盤、35 点）— EWS < 40 かつ資金繰りギャップ ≥ 0。
- **適時適切な情報開示**（情報開示、25 点）— 12 ヶ月分の試算表 + TDB スコア + エラーなし。

ステートへの出力：`hosho_kaijo_score`、`hosho_kaijo_conditions`、
`hosho_kaijo_eligible`（= スコア ≥ `HOSHO_ELIGIBLE_SCORE`）、`succession_ready`。
既知の弱点：*分離*プロキシが FSA の意図と逆に読める場合があります。要改善。

## 機能 2 — マルチクリティック「主幹事銀行（取りまとめ）」（債権者会議）

3 つの**独立した並列クリティックノード**（互いの評価を参照してはならない）が、それぞれ決定論的な
`CriticFeedback`（合否 + ブロッカー + 優先度）を出力します。

- **主幹事銀行**（P1、説明責任）：`yakuin_hoshu_cut` がなければ FAIL。赤字の場合は
  `personal_asset_disposal` も必要。これらは**バンカー専用**フラグです。
- **協調融資銀行**（P2、公平性）：負担が `PRO_RATA_TOLERANCE` 内で持分比率に応じていなければ FAIL
  （`lender_stakes` が設定されている場合は持分ベース、そうでなければアップリフトプロキシ）。
- **信用保証協会**（P0、コンプライアンス）：3〜5 年の信頼できる回収経路がなければ FAIL。

`lead_arranger` がこれらを集約します（取りまとめ）：**いずれかが FAIL → 否決**、
**全員が PASS → 承認**。ブロッカーは P0>P1>P2 の順に並べられ、決定論的な負担分担表も生成されます。
唯一のブロッカーがバンカー専用の場合は `needs_human` を出力し、グラフはストラテジストをエスカレーションに
ループさせる代わりに HITL にルーティングします。

**リデューサーの注意点：** `critic_feedbacks` は `CRITIC_FEEDBACKS_CLEAR` センチネルを使用する
カスタムリデューサーを使います。`strategist_node` は修正ラウンド間でリセットするためにセンチネルを返します
（単純な `[]` は no-op となり古い判定が蓄積されます。これは実際に発生したバグです。再導入しないこと）。

## ステートチートシート（`app/backend/state.py`）

よく触れる主要フィールド：`shisanhyo`、`working_capital_gap`、`ews_score`、
`fsa_classification`、`proposed_strategies`、`critic_feedbacks`、
`negotiation_status`（`pending|approved|rejected|needs_human`）、

`yakuin_hoshu_cut` / `personal_asset_disposal`、および経営者保証の出力フィールド。

## 規約とガードレール

- **閾値の唯一の正解：`app/shared/constants.py`。** インポートして使用すること。
  ローカルコピーを再定義しないこと（ドリフトバグが 2 度発生しています）。
- 長い日本語プロンプトテキストは Python ファイルではなく `app/backend/prompts/` に追加すること。
- モックデータは `MockDataProvider`（`app/backend/tools/`）の背後で差し替え可能です。
  実際の TDB・コアバンキング・EDINET クライアントは同じインターフェースの背後に差し込めます。
  グラフ変更不要。
- フィクスチャはパッケージ内の `app/backend/tools/fixtures/` に同梱されています。

## 作業方法

```bash
make setup     # uv install, .env, containers, seed DB
make run-dev   # run the stack
make verify    # ruff + mypy --strict + pytest  <- マージ前に必ずパスさせること
make test      # pytest only
```

`make verify` が契約です。CI（`.github/workflows/ci.yml`）は同じゲート（ruff + mypy --strict
+ pytest）を、`main` への全プッシュと全プルリクエストで実行します。
エージェントサンドボックスはインストール用のネットワークを持たないため、
**パイプラインが権威あるチェック**です。マージ前に必ずグリーンにしてください。

## 計画中の機能（Part 4）— マルチエージェント債権者会議*シミュレーター*

**ステータス：仕様策定済み（未実装）。追加前に必ず読むこと。**

### なぜ必要か

現在の債権者会議は完全に決定論的です。3 つの純粋関数クリティックが合否を判定し、
`lead_arranger` が集約（取りまとめ）します。これは監査可能性を守りますが、
*実際の*会議は交渉であり、担当バンカーは主幹事銀行・協調融資銀行・信用保証協会が
どのように主張するかを事前に把握した上で臨む必要があります。
Part 4 では、エンジンがバンカーの**リハーサルツール**となるよう、
その会議をシミュレートする**LLM ペルソナレイヤー**を追加します。
バンカーが唯一の真の意思決定者であることは変わりません。

### 唯一のルールは引き続き適用されます（破らないこと）

**決定論的ゲートはすべての判定と数値の唯一の正解であり続けます。**
新しいエージェントは*スタンス、議論、実現可能性*について推論しますが、
合否ゲートや数値については推論しません。具体的には：

- `CriticFeedback.status`（合否）と構造化されたブロッカーコード
  （`yakuin_hoshu_not_cut`、`no_asset_disposal` など）は 100% ルールベースのままです。
  `lead_arranger` の `needs_human` ルーティングがこれらを文字列マッチングするためです。
  LLM がこれらを生成・変更してはなりません。
- LLM は新しい*アドバイザリー*フィールドのみを追加します（シミュレートされた議論、
  予測される譲歩、実現可能性の意見）。これらはルーティングやスコアリングに影響しません。
- すべてのエージェントは**決定論的なオフラインフォールバック**を持たなければなりません
  （`polish_keikakusho` を参考に：LLM が設定されていない場合は no-op または定型出力）。
  これにより、ネットワークのない CI サンドボックスでも `make verify` がグリーンを保ちます。

### アーキテクチャ：ハイブリッドエージェント（ゲート + ペルソナ）

各クリティックは**ハイブリッド**になります。既存の決定論的ゲートは変更されず、
その上に新しいペルソナレイヤーが推論を加えます。

```
strategist
   │
   ▼  (NEW, upstream pre-screen)
feasibility_critic  ← TRUE agent: loops over each proposed Strategy, optionally
   │                  RAG over past plans / industry benchmarks, asks "is this
   │                  operationally achievable for THIS firm?" Emits advisory
   │                  feasibility notes. Does NOT gate; annotates strategies.
   ├─fan-out─▶ main_bank_critic   (gate PASS/FAIL  +  persona arg — 主幹事銀行 voice)
   ├─fan-out─▶ sub_bank_critic    (gate PASS/FAIL  +  persona arg — 協調融資銀行 voice)
   └─fan-out─▶ guarantor_critic   (gate PASS/FAIL  +  persona arg — 信用保証協会 voice)
              │
              ▼
        lead_arranger  ← becomes ORCHESTRATOR/chair-agent: consolidates the
                          deterministic verdicts (unchanged) AND assembles a
                          *meeting briefing* from the persona arguments — the
                          rehearsal the banker reads before HITL.
              │
              ▼
        hitl_negotiation (human — the only real decider)
```

**役割の分離（明確に区別すること）：**
- `feasibility_critic` = 上流。「この企業は実際にそれを実行できるか？」（業務的観点）を問う。
- 3 つのクリティック = 「これは融資機関間で公平・準拠・説明責任があるか？」（ペルソナ別ゲート + シミュレートされたスタンス）。
- `lead_arranger` = 議長：判定を集約し、負担分担表（変更なし）**および**新しいシミュレート会議ブリーフィングを生成。

### 実装上の注意

- **新モジュール：** `app/backend/nodes/critics/feasibility.py`（`feasibility_critic_node`）。
  `strategist → feasibility_critic → {three critics}` と配線します
  （既存のファンアウトの前に挿入。ファンアウトエッジは `feasibility_critic` から発生するよう移動）。
- **プロンプト**は `app/backend/prompts/` に配置します（ペルソナごとに 1 つ、静的 `.md` ファイル）。
  Python ファイルにインラインで書かないこと。既存の `extraction_rules.md` / `kaizen_templates.md` を参照。
  これにより、`NEXT_STEPS.md` の将来のプロンプトレジストリ移行も準備されます。
- **LLM クライアント：** `kaizen_generation.py` の `polish_keikakusho` パターンを再利用します
  （`httpx` 経由の OpenAI 互換チャット補完、`Settings.llm_*`、フォールバック付きベストエフォート）。
  コードが増えたら小さな共有ヘルパーに切り出すこと。
- **ステート（追加のみ — 既存フィールドの削除・リネーム禁止）：**
  - `feasibility_notes: list[dict]`（戦略ごとのアドバイザリー。並列生成の場合は `critic_feedbacks` 同様のリデューサー）。
  - `CriticFeedback` にオプションの `simulated_argument: str = ""` を追加
    （デフォルト値により既存のテスト・シリアライゼーションのバイト安定性を維持）。
  - `meeting_briefing: str | None`（`lead_arranger` からのリハーサルテキスト）。
  - HITL の `_interrupt_payload` に `meeting_briefing` と実現可能性を表示。
- **決定論性ガード：** LLM が設定されていない状態で、ゲート、ブロッカーコード、ルーティング、
  負担分担表が今日と完全に同一であることを確認するテストを追加すること
  （ペルソナ・ブリーフィングフィールドは空・定型）。これが命題が成立することを証明する回帰テストです。
- **ショーケース価値：** これが正直な「マルチエージェントシステム」のストーリーです。
  決定論的な規制スパインを、人間のバンカーをリハーサルするマルチエージェント*シミュレーション*が
  包み込む構造です。代替するのではなく、リハーサルします。

### 推奨実装順序
1. `CriticFeedback` にオプションの `simulated_argument` を追加（動作変更なし）。
2. `feasibility_critic` エージェント + プロンプトを追加（アドバイザリーのみ、オフラインフォールバック付き）。
3. 3 つのクリティックにペルソナレイヤーを追加（ゲートは変更なし。議論を追加）。
4. `lead_arranger` を `meeting_briefing` を出力するよう更新。HITL に表示。
5. 決定論性パリティテスト + オフライン `make verify` がグリーンを維持することを確認。

## 注意すべき落とし穴

- **バンカー専用**ブロッカーでストラテジストをループさせないこと → `needs_human` を使用。
- `critic_feedbacks` のリセットに `[]` を返さないこと → `CRITIC_FEEDBACKS_CLEAR` を使用。
- LLM の出力を数値や判定に追加しないこと。
- `render_keikakusho` の正確な出力を壊したり、ステートフィールドを削除したりしないこと。
- クリティックを互いに独立させ、決定論的に保つこと。
